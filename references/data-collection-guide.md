# 数据收集技术方案

> 本文档解释我们如何从小红书获取数据，以及背后的技术原理。

---

## 一、为什么需要逆向？

### 小红书网页版的限制

当你在小红书网页版搜索时：

```
搜索页（已登录）        →  可以看到笔记列表（标题、作者、点赞数）
笔记详情页              →  "当前笔记暂时无法浏览，请打开App扫码查看"
评论区                  →  网页版不显示或需要App验证
```

**结论**：网页版只能看到"橱窗"，看不到"内容"。

### 网页版 vs API 能拿到的数据对比

| 数据 | 网页版搜索 | 网页版详情 | API 逆向 |
|------|:---------:|:---------:|:-------:|
| 笔记标题 | ✅ | ❌ | ✅ |
| 正文/描述 | ❌ | ❌ | ✅ |
| 点赞数 | ✅ | ❌ | ✅ |
| 收藏数 | ❌ | ❌ | ✅ |
| 评论数 | ❌ | ❌ | ✅ |
| **评论内容** | ❌ | ❌ | ✅ |
| **评论回复** | ❌ | ❌ | ✅ |
| 标签 | ❌ | ❌ | ✅ |
| 发布时间 | ✅ | ❌ | ✅ |
| 作者信息 | ✅ | ❌ | ✅ |

## 二、技术原理：小红书 API 签名机制

### 2.1 为什么直接调 API 不行？

小红书的所有 API 请求都需要**签名验证**。如果没有正确签名，服务器直接拒绝：

```http
GET /api/sns/web/v1/feed
Headers:
  x-s: xxxxxx          ← 没有 → 403 Forbidden
  x-t: 1714648800     ← 没有 → 403 Forbidden
  x-s-common: xxxxxx  ← 没有 → 403 Forbidden
```

### 2.2 签名怎么生成的？

签名算法是小红书前端 JS 代码里的一段逻辑，被安全研究者**逆向提取**出来了。

```
输入：
  ├── a1 cookie 值（从登录态中提取）
  ├── API 路径（如 /api/sns/web/v1/feed）
  ├── 请求参数（如 {source_note_id: "xxx"}）
  └── 当前时间戳
        ↓
执行：xhs_main_260411.js 中的签名函数
        ↓
输出：
  ├── x-s: 主签名
  ├── x-t: 时间戳
  └── x-s-common: 通用签名
```

### 2.3 我们用的工具：Spider_XHS

**Spider_XHS**（GitHub: cv-cat/Spider_XHS）是一个开源项目，它：

1. 内置了逆向提取的签名 JS 文件（`static/xhs_main_260411.js`）
2. 用 Python 的 `PyExecJS` 调用 JS 函数生成签名
3. 用 `requests` 发送带签名的 HTTP 请求
4. 解析返回的 JSON 数据

```
你的 Cookie
    │
    ▼
Spider_XHS (Python)
    ├── 读取 .env 中的 COOKIES
    ├── 解析出 a1 字段
    ├── 调用 JS 签名函数（PyExecJS）
    ├── 生成 x-s, x-t, x-s-common
    ├── 发送 HTTP 请求（requests）
    └── 解析 JSON 响应
```

## 三、安装与配置步骤

### 步骤1：环境要求

```bash
# 检查版本
python3 --version   # 需要 3.10+
node --version      # 需要 20+
```

### 步骤2：下载 Spider_XHS

```bash
# 方式A：git clone（如果网络通畅）
git clone --depth 1 https://github.com/cv-cat/Spider_XHS.git ~/Spider_XHS

# 方式B：手动下载 ZIP
# 打开 https://github.com/cv-cat/Spider_XHS
# 点击 Code → Download ZIP
# 解压到 ~/Spider_XHS
```

### 步骤3：安装依赖

```bash
cd ~/Spider_XHS

# Python 依赖
pip3 install -r requirements.txt

# Node.js 依赖
npm install
```

### 步骤4：获取 Cookie

先提醒一句：

- 必须先登录小红书网页版
- 必须使用小红书小号，严禁主号
- 不要把 `browser_state.json`、`.env` 或 Cookie 截图上传到公开仓库

**方法A：自动提取（推荐）**

用我们之前写的浏览器自动化脚本登录小红书小号，然后运行：

```bash
python3 ~/Spider_XHS/get_cookie.py
```

这个脚本会从 `browser_state.json` 中提取 Cookie 并写入 `.env`。

**方法B：手动提取**

```
1. 用 Chrome 登录小红书网页版（https://www.xiaohongshu.com）
2. 按 F12 打开开发者工具
3. 切换到 Network（网络）标签
4. 刷新页面，找到任意一个请求（如 feed）
5. 在请求头中找到 cookie 字段
6. 复制整段 cookie 字符串
7. 写入 ~/Spider_XHS/.env：
   COOKIES='你的cookie字符串'
```

### 步骤5：验证安装

```bash
cd ~/Spider_XHS
python3 -m spider.spider
```

如果看到 `msg: 成功`，说明配置正确。

## 四、安全爬虫脚本

我们写了一个安全版脚本 `safe_spider.py`，特点：

### 限速策略

```python
# 每篇笔记间隔 3-6 秒
safe_delay(3, 6)

# 每个关键词搜索后休息 10-15 秒
safe_delay(10, 15)

# 每篇笔记最多爬 20 条评论
max_comments = 20

# 每个关键词最多分析 10 篇笔记
max_notes = 10
```

### 运行命令

```bash
cd ~/Spider_XHS

# 搜索单个关键词，分析10篇笔记，每篇20条评论
python3 safe_spider.py -k "毕业生求职" -n 10 -c 20

python3 safe_spider.py -k "毕业生求职" "面试技巧" "谈薪话术" -n 10 -c 20
```

## 五、数据安全与风险

### 封号风险等级

| 行为 | 风险等级 | 说明 |
|------|:-------:|------|
| 搜索 + 看列表 | ⭐ 低 | 类似正常用户行为 |
| 获取笔记详情 | ⭐⭐ 中 | 需要签名，但频率低安全 |
| 获取评论 | ⭐⭐ 中 | 同上 |
| 批量下载视频/图片 | ⭐⭐⭐ 高 | 大量媒体请求易触发风控 |
| 高频请求（1秒多次） | ⭐⭐⭐⭐⭐ 极高 | 必死 |

### 我们的安全措施

```
✅ 限速：每篇笔记 3-6 秒间隔
✅ 限量：每关键词最多 10-20 篇
✅ 用小号：不危及主账号
✅ 本地运行：不用服务器IP
✅ 仅只读：不发帖、不点赞、不评论
```

### 如果被封号怎么办？

- 小号被封：影响不大，重新注册一个
- 主号被封：风险极高，务必**只用小号**跑爬虫

## 六、两种数据获取方式对比

| 维度 | 浏览器自动化 | API 逆向（Spider_XHS） |
|------|:-----------:|:---------------------:|
| **安装难度** | 低 | 中 |
| **数据完整性** | 只能拿搜索页 | 能拿详情+评论 |
| **速度** | 慢（渲染页面） | 快（纯HTTP） |
| **稳定性** | 高（不依赖签名算法） | 中（签名算法可能过期） |
| **封号风险** | 低 | 中 |
| **维护成本** | 低 | 高（需跟进签名更新） |
| **适用场景** | 日常选题研究 | 深度竞品分析 |

### 建议的使用策略

```
日常选题研究（每天）
    └── 用浏览器自动化（search脚本）
        └── 搜关键词 → 拿标题/点赞/作者 → 分析选题

深度竞品分析（每周1-2次）
    └── 用 API 逆向（safe_spider）
        └── 搜关键词 → 拿正文 → 拿评论 → 拆爆款
```

## 七、故障排查

### 问题1：Cookie 失效

```
症状：所有请求返回失败，msg: '登录已过期'
解决：
  1. 重新登录小红书网页版小号
  2. 提取新 Cookie
  3. 更新 .env 文件
```

### 问题2：签名错误

```
症状：403 Forbidden 或 "签名验证失败"
原因：小红书更新了前端 JS，签名算法变了
解决：
  1. 更新 Spider_XHS 到最新版
  2. 替换 static/ 目录下的 JS 文件
```

### 问题3：请求超时

```
症状：连接超时或无响应
解决：
  1. 检查网络连接
  2. 增加延时：修改 safe_delay(5, 10)
  3. 减少并发：减少 max_notes
```

### 问题4：PyExecJS 报错

```
症状：execjs._exceptions.RuntimeError
原因：Node.js 版本不兼容或安装不完整
解决：
  1. 重装 Node.js（推荐 20.x LTS）
  2. 确保 node 命令在 PATH 中
  3. 重装 PyExecJS: pip install --force-reinstall PyExecJS
```

## 八、总结

**数据获取的完整链路**：

```
你的需求
    │
    ├── 日常选题研究？
    │       └── 浏览器自动化（已验证，稳定）
    │           └── 拿标题/作者/点赞 → 分析选题
    │
    └── 深度竞品拆解？
            └── API 逆向 Spider_XHS（已配置）
                └── 拿正文/评论/标签 → 拆爆款
```

**核心原则**：
1. 用小号，不用主号
2. 限速，不贪多
3. 本地运行，不用服务器
4. 只读不写，低调使用
